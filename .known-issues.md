# Known Issues — Accepted, Not Fixing

This file lists issues that reviewers have previously flagged but the team has explicitly decided not to fix, along with the reason. Code reviewers (`/review-all`, `/simplify`) should consult this file and skip issues listed here.

When an issue is fixed and no longer applies, remove it from this file.

---

## Infrastructure Constraints

### Rate limiter is in-memory (not Redis-backed)
**Why accepted:** Single-instance EC2 deployment. Adding Redis for horizontal scaling is not justified at current traffic. Cloudflare WAF provides the primary rate-limiting backstop.
**Revisit when:** Moving to multi-instance / PM2 cluster mode, or traffic exceeds ~100 req/min sustained.
**File:** `lib/rate-limiter.ts`

### DNS Rebinding TOCTOU window
**Why accepted:** Fix requires IP pinning (substituting resolved IP into URL with `Host` header override). Non-trivial and low exploitation value at current threat model. SSRF check + redirect re-validation already block the common attack paths.
**Revisit when:** Handling requests from untrusted tenants, or SSRF becomes a demonstrated attack vector.
**File:** `lib/security.ts:190-250`

### CSP `script-src 'unsafe-inline'`
**Why accepted:** Next.js emits inline `<script>` for JSON-LD (`dangerouslySetInnerHTML` in `layout.tsx`). Fix requires nonce-based CSP middleware, which historically has caused hydration issues in Next.js App Router. Risk-vs-reward unfavorable for a site with no untrusted HTML rendering.
**Revisit when:** Upgrading to a Next.js version with stable nonce support, or adding any user-generated HTML rendering.
**File:** `next.config.ts:36`

### `safeFetch` DNS lookup not signal-aware
**Why accepted:** Caller's `AbortSignal` fires an outer timeout, but in-flight DNS lookups inside `validateDnsResolution` keep running up to 3s after abort. Resource impact is small (at most 3 zombie lookups on timeout). Fix requires threading signal through `dns.promises.lookup` which has no native support.
**Revisit when:** Observing DNS resolver saturation in production metrics.
**File:** `lib/security.ts:232-243`

### X-Forwarded-For rightmost-IP trust
**Why accepted:** Deployment enforces Cloudflare-only ingress at EC2 Security Group level (ports 80/443 allow Cloudflare IPs only). `CF-Connecting-IP` is always present in production. Unknown-IP fallback uses shared bucket as defense-in-depth.
**Revisit when:** Adding direct-origin access paths, or removing Cloudflare.
**File:** `middleware.ts:88-95`

## Architectural Choices (Not Bugs)

### Module-level `let` for scroll lock counter
**Why accepted:** Simpler than context provider for a single modal pattern. Safety valve added for Fast Refresh / Strict Mode edge cases.
**Revisit when:** Adding nested modals or more than 2 components using body-scroll lock.
**File:** `components/features/results/components/check-references.tsx:11-13`

### `useAbortController.getSignal` not memoized with `useCallback`
**Why accepted:** Current callers read `getSignal` synchronously inside event handlers — referential instability has no observable effect. Only relevant if passed as a React effect dependency.
**Revisit when:** Adding a consumer that lists `getSignal` in `useEffect` / `useCallback` deps.
**File:** `lib/hooks/useAbortController.ts:17`

## Guidelines

- **Be specific about why.** "Too hard" is not a reason. State the actual constraint.
- **Be specific about revisit condition.** Not "someday" — a concrete signal ("traffic exceeds X", "adopt Y feature").
- **Keep entries current.** Delete when fixed. Update if the reason changes.
- **Reviewers must respect this.** If you disagree with an entry, raise it with the team — don't just re-flag it.
